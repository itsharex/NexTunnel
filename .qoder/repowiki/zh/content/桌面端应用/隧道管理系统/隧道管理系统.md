## 简介
本技术文档面向 NexTunnel 隧道管理系统，聚焦于客户端隧道管理器的设计与实现，涵盖以下主题：
- 隧道生命周期：创建、启动、停止、销毁
- 动态隧道管理：运行时新增/删除隧道
- 运行时配置更新与状态监控
- 自动重连机制：指数退避与抖动、心跳保活
- 配置加载、校验与持久化（SQLite）
- 跨平台TUN网络抽象层：统一的虚拟网络设备接口
- IPAM集成：IP地址分配与路由管理
- API 使用与状态查询示例路径
- 性能优化建议与故障排除

## 项目结构
NexTunnel 采用"桌面端 + 服务端"的双端架构，桌面端使用 Wails（Go + Vue），通过 Wails 绑定暴露 Go 方法给前端调用；服务端提供中继与控制平面能力。本文重点覆盖桌面端隧道客户端子系统，并新增跨平台TUN网络抽象层和IPAM集成。

## 核心组件
- 隧道管理器（Manager）：顶层编排者，负责连接服务器、注册隧道、处理消息、心跳、动态增删隧道、状态聚合与导出。
- 控制客户端（ControlClient）：维护到服务器的持久控制通道，负责认证、收发消息、读循环、线程安全发送。
- 隧道实例（Tunnel）：单个隧道的运行实体，负责工作连接建立、数据桥接、状态与流量统计。
- 重连器（Backoff）：指数退避+抖动策略，支持重试计数与上下文取消。
- 配置（TunnelClientConfig/TunnelDef）：客户端配置与单隧道定义，含默认值与HTTP扩展字段。
- 本地配置存储（SQLite）：隧道配置与应用设置的持久化，提供CRUD与迁移。
- **新增** TUN网络抽象层：跨平台虚拟网络设备抽象，统一Linux/macOS/Windows的TUN接口。
- **新增** IPAM管理：IP地址分配与路由管理，支持虚拟网络的IP地址规划与路由配置。

## 详细组件分析

### 隧道管理器（Manager）
职责与行为
- 初始化：填充默认配置（客户端ID、重连基线/最大延迟、心跳间隔），预建隧道实例。
- 启动：通过指数退避循环尝试连接服务器，成功后注册所有隧道并启动心跳。
- 消息处理：处理服务器的心跳请求/响应、工作连接请求等。
- 动态管理：AddTunnel/RemoveTunnel 支持运行时增删，并在已连接时通知服务器。
- 状态聚合：GetStatus 返回各隧道实时状态与流量统计。
- 停止：优雅关闭，取消上下文，关闭控制连接，标记隧道为非活跃。
- **新增** IPAM集成：在隧道启动前分配虚拟网络IP地址，确保网络隔离。

### 自动重连机制（指数退避+抖动）
- 配置项：基线延迟、最大延迟、乘数、抖动比例。
- 行为：每次失败计算下一次延迟，上限裁剪，加入随机抖动；成功则重置计数。
- 集成点：Manager.Start 内部使用 Backoff.Run 包裹 connectAndRun，实现断线自动重连。

```mermaid
flowchart TD
Enter(["开始重连循环"]) --> Attempt["尝试连接/注册"]
Attempt --> Success{"成功?"}
Success -- 否 --> Delay["计算延迟=指数退避+抖动"]
Delay --> Sleep["等待延迟"]
Sleep --> Attempt
Success -- 是 --> Reset["重置计数"]
Reset --> Exit(["继续主循环"])
```

### API 使用与状态查询（示例路径）
- 获取隧道列表：前端调用 GetTunnels → Wails 绑定 → app.GetTunnels → 从 Store 读取并合并 Manager 的实时状态。

- 连接状态：GetConnectionStatus → Manager.IsConnected。
- 流量统计：GetTrafficStats → 聚合 Manager.GetStatus 中的 BytesIn/BytesOut。

### 平台特定实现

#### Linux内核TUN实现
Linux平台使用/dev/net/tun字符设备，通过ioctl系统调用创建TUN设备：

- **设备创建**：使用TUNSETIFF ioctl配置设备类型为TUN，禁用包头信息。
- **IP配置**：支持通过系统工具或netlink库配置IP地址和子网掩码。
- **权限要求**：需要root权限或适当的CAP_NET_ADMIN能力。

#### macOS内核TUN实现
macOS使用utun内核接口，通过4字节的协议族头部处理IPv4/IPv6：

- **协议头处理**：根据IP版本自动添加相应的AF_INET或AF_INET6头部。
- **设备命名**：系统自动分配utun设备名称（如utun3）。
- **权限要求**：通常需要管理员权限。

#### Windows内核TUN实现
Windows平台使用Wintun驱动程序，提供高性能的用户态TUN实现：

- **Wintun驱动**：基于Wintun.dll的高性能TUN实现。
- **会话管理**：通过Wintun会话对象管理数据包的发送和接收。
- **权限要求**：需要管理员权限安装驱动。

#### 用户态netTun实现
作为回退方案，提供基于Go通道的虚拟TUN设备：

- **测试用途**：主要用于单元测试和开发验证。
- **数据流**：通过goroutine通道实现数据包的异步传输。
- **限制**：仅支持本地测试，不适用于生产环境。

## IPAM集成与网络管理

### IP地址管理（IPAM）架构
NexTunnel集成了完整的IP地址管理（IPAM）系统，为虚拟网络提供IP地址分配和路由管理能力：

- **Allocator接口**：定义了IP地址分配的标准接口，支持分配、释放、查询和列出所有分配。
- **IPAM实现**：基于CIDR子网的IP地址分配器，支持网关地址保留和循环分配算法。
- **线程安全**：使用互斥锁保护并发访问，确保多线程环境下的数据一致性。

### IP地址分配算法
IPAM实现了高效的IP地址分配算法，确保虚拟网络的IP地址合理分配：

- **CIDR支持**：支持任意CIDR块的IP地址池管理。
- **网关保留**：自动保留子网的网络地址、广播地址和网关地址。
- **循环分配**：从下一个可用主机地址开始分配，支持循环回到开头。
- **冲突检测**：在分配前检查现有分配，避免IP地址冲突。

分配流程：
1. 检查节点是否已有分配的IP地址
2. 收集所有已使用的IP地址（包括网络、广播、网关）
3. 从nextHost开始查找第一个可用的IP地址
4. 如果到达子网末尾，循环到开头继续查找
5. 分配找到的IP地址并更新nextHost

### 路由管理（RouteManager）
路由管理器提供了虚拟网络的路由表管理功能：

- **路由条目**：每个路由条目包含目标网络、网关、网络接口和度量值。
- **最佳匹配**：支持根据目标IP地址查找最佳匹配的路由。
- **重复检测**：防止添加重复的路由条目。
- **线程安全**：使用读写锁支持高并发的路由表操作。

路由查找算法：
1. 遍历所有路由条目
2. 检查目标网络是否包含指定的IP地址
3. 选择具有最长前缀匹配的路由
4. 返回最佳匹配的路由条目

## 性能考虑
- 并发与锁
  - Manager 对隧道集合使用读写锁，避免频繁加锁；状态查询与动态增删均受控。
  - **新增** IPAM使用互斥锁保护分配操作，确保线程安全。
- 数据桥接
  - Tunnel 使用 io.Copy 双向复制，WaitGroup 保证收尾；桥接阶段不进行额外解析，减少 CPU 开销。
  - **新增** TUN设备实现零拷贝数据包传输，减少内存分配。
- 心跳与重连
  - 心跳周期可配置，默认 30 秒；指数退避带抖动，避免雪崩效应。
- 存储
  - SQLite 启用 WAL 模式提升并发；Store 使用参数化 SQL 防注入。
- **新增** TUN性能优化
  - 用户态netTun使用缓冲通道，支持批量数据包处理。
  - 内核TUN直接通过系统调用传输，避免用户态复制开销。
- **新增** IPAM性能优化
  - 使用位图标记已用IP，提高冲突检测效率。
  - 支持IP地址池预分配，减少运行时分配开销。

## 故障排除指南
- 无法连接服务器
  - 检查 ServerAddr 是否正确；确认网络可达；查看 ControlClient 的认证响应与错误日志。
  - 关注 Manager 的自动重连是否生效（日志中应有"连接失败"、"重试"等记录）。
- 隧道未激活
  - 确认注册流程已完成（NewProxyResp 成功）；检查 Manager.handleServerMessage 是否收到回执。
  - 查看 Tunnel.Info 的状态是否为 active。
- 工作连接失败
  - 检查本地服务地址与端口是否可达；确认 StartWorkConn 到 WorkConn 的握手链路。
- 断线重连
  - 观察指数退避日志；确认 ctx 未提前取消；确保心跳未被阻塞。
- 配置持久化问题
  - 确认 SQLite 文件存在且可写；检查迁移是否成功；使用 Store 的 CRUD 方法验证。
- 前端状态不同步
  - 确认 Wails 绑定正常；检查 Pinia Store 的刷新逻辑与错误捕获。
- **新增** TUN设备问题
  - 检查平台支持情况：使用CurrentPlatform()确认内核TUN支持。
  - 权限问题：Linux需要root权限，Windows需要管理员权限。
  - 设备冲突：确认TUN设备名称唯一性，避免与其他VPN软件冲突。
- **新增** IPAM分配问题
  - 子网过大：确保子网至少为/30，保证有足够的可用IP地址。
  - 地址耗尽：检查IPAM.ListAllocations()确认是否有地址可用。
  - 网关冲突：确认网关地址在子网范围内且未被分配。

## 结论
